# 代表処理シナリオ

## 概要

第2段階で実装した代表オンライン処理の実行シナリオを整理します。対象は、在庫照会、入荷登録、出荷指示登録の3処理です。

---

## 1. 在庫照会シナリオ

### 正常系

1. 利用者が商品コード、倉庫コード、ロケーションコードを指定する
2. システムが在庫を検索する
3. 該当在庫が存在する場合、現在庫数量を返す
4. 条件未指定または部分指定の場合は、該当範囲の一覧を返す

### 主な異常系

| ケース | 挙動 |
|------|------|
| 在庫が存在しない | 空一覧を返し、呼び出し元で 0 件として扱う |
| 商品名が取得できない | 商品コードは返し、商品名は `(unknown item)` とする |

### 業務的意味

- 倉庫担当者が現在庫を即時に確認するための基本処理
- 後続の入荷、出荷、棚卸差異確認の前提情報になる
- 更新を伴わないため、オンライン処理の中でも最も純粋な参照責務を持つ

---

## 2. 入荷登録シナリオ

### 正常系

1. 利用者が入荷番号、商品コード、倉庫コード、ロケーションコード、数量、入荷日を指定する
2. システムが商品、倉庫、ロケーションの存在を検証する
3. システムが入荷実績を `InboundReceipt` として登録する
4. 該当在庫が存在する場合は数量を加算する
5. 該当在庫が存在しない場合は新規在庫を作成する
6. 更新後の在庫数量を返す

### 主な異常系

| ケース | 挙動 |
|------|------|
| 商品未存在 | エラーとして処理を中断する |
| 倉庫未存在 | エラーとして処理を中断する |
| ロケーション未存在 | エラーとして処理を中断する |
| 数量が 0 以下 | エラーとして処理を中断する |

### 業務的意味

- 在庫増加の責務境界を明確に示す代表処理
- 入荷実績の保持と在庫更新が同一ユースケース内で連続して発生する
- 将来の保証単位研究では、`InboundReceipt` 登録と `Stock` 更新の一貫性境界として扱いやすい

---

## 3. 出荷指示登録シナリオ

### 正常系

1. 利用者が出荷指示番号、商品コード、倉庫コード、ロケーションコード、指示数量、出荷予定日、出荷先コードを指定する
2. システムが商品、倉庫、ロケーション、出荷先の存在を検証する
3. システムが `OutboundOrder` を `Planned` 状態で登録する
4. 在庫数量は変更しない
5. 登録した出荷指示情報を返す

### 主な異常系

| ケース | 挙動 |
|------|------|
| 商品未存在 | エラーとして処理を中断する |
| 倉庫未存在 | エラーとして処理を中断する |
| ロケーション未存在 | エラーとして処理を中断する |
| 出荷先未存在 | エラーとして処理を中断する |
| 数量が 0 以下 | エラーとして処理を中断する |

### 業務的意味

- 将来の引当、出荷確定につながる未確定イベントを表現する
- 在庫を減らさず、業務イベントだけを保持するため、在庫更新責務とイベント登録責務を分離しやすい
- COBOL模倣時には、オンライン入力プログラムとして独立させやすい

---

## 実装上の位置づけ

| 処理 | 実装クラス | 主な更新対象 |
|------|------------|--------------|
| 在庫照会 | `GetStockUseCase` | なし |
| 入荷登録 | `RegisterInboundUseCase` | `InboundReceipt`, `Stock` |
| 出荷指示登録 | `RegisterOutboundOrderUseCase` | `OutboundOrder` |
